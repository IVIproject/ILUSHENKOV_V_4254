# Раздел «РАЗРАБОТКА» для дипломной работы по проекту `ai-servise`

> Этот материал можно вставлять в пояснительную записку как основу главы «Разработка», а также использовать как дорожную карту доработки проекта перед защитой.

## 1. Цель и постановка задачи разработки

Цель разработки — создать прикладной AI‑сервис для задач хостинг/доменного бизнеса, который:

- предоставляет единый API для нескольких сценариев генерации;
- поддерживает как локальные LLM (через Ollama), так и внешние модели (через OpenRouter/OpenAI‑совместимый интерфейс);
- ведёт историю запросов и технические метрики для аудита;
- включает web‑кабинет (gateway) для пользователей и администратора;
- развёртывается в контейнерной среде Docker Compose.

Практически система решает задачи:

1. генерация общего текста (`/generate`);
2. потоковая генерация (`/generate/stream`);
3. генерация доменных имен (`/generate/domains`);
4. FAQ‑поддержка на основе импортируемой базы вопросов/ответов (`/support/...`);
5. генерация PHP‑страницы по шаблону (`/page-template/generate-file`);
6. централизованная работа с каталогом моделей и тарификацией в gateway.

## 2. Обоснование выбора технологий

### 2.1 Backend и API

Основой выбран **FastAPI**: асинхронная web‑платформа с встроенной OpenAPI‑документацией, валидацией входных данных (Pydantic) и удобной структурой маршрутов. В проекте это видно по большому набору REST‑эндпоинтов в `app/main.py`.

### 2.2 Хранение данных

Используется **PostgreSQL** как транзакционная СУБД для:

- логов запросов/ответов (`request_logs`);
- базы FAQ (`support_faq_entries`);
- метрик качества FAQ‑ответов (`support_faq_query_metrics`);
- пользователей gateway, каталога моделей, логов потребления токенов.

ORM‑слой реализован через SQLAlchemy‑модели в `app/models.py`, миграции — Alembic.

### 2.3 Интеграция с моделями

- **Ollama** используется для локальных моделей (низкая зависимость от внешних API, повышенная приватность).
- **OpenRouter** подключается через OpenAI‑совместимый HTTP‑клиент, что расширяет доступный каталог моделей без переработки прикладной логики.

### 2.4 Развёртывание

Связка **Nginx + FastAPI + PostgreSQL** разворачивается через `docker-compose.yml`, где Nginx выступает reverse proxy (порт 8080).

## 3. Архитектура и логика взаимодействия компонентов

### 3.1 Архитектурная схема

Текущая схема:

1. Клиент отправляет HTTP‑запрос на Nginx.
2. Nginx проксирует в FastAPI‑приложение.
3. FastAPI валидирует payload Pydantic‑схемами.
4. В зависимости от режима вызывает Ollama или OpenAI‑совместимый прокси.
5. Результат сохраняется в PostgreSQL.
6. Клиент получает JSON/stream‑ответ.

### 3.2 Основные модули backend

- `app/main.py` — маршруты API, gateway‑функционал, OpenAI‑совместимые эндпоинты.
- `app/services.py` — бизнес‑логика режимов генерации, FAQ‑ранжирование, парсинг.
- `app/gateway_services.py` — модельный каталог, токен‑оценка, расчёт стоимости, хеширование паролей/API‑ключей, вызов внешнего провайдера.
- `app/models.py` — структуры таблиц БД.
- `app/schemas.py` — контракты запросов/ответов.
- `app/page_templates.py` — логика шаблонизации и генерации PHP‑страниц.

### 3.3 Логирование и трассировка

Через middleware каждому запросу назначается `X-Request-ID`. Идентификатор возвращается клиенту и используется для диагностики в логах.

## 4. Реализация функциональных подсистем

### 4.1 Подсистема базовой генерации

Эндпоинт `/generate` выполняет классический prompt->response сценарий, записывает результат в журнал запросов. Потоковый вариант `/generate/stream` отдаёт NDJSON‑чанки и завершает маркером `done=true`.

### 4.2 Подсистема генерации доменных имен

`/generate/domains` формирует строгое промпт‑ограничение (по зоне, количеству, формату), затем валидирует и нормализует домены. Если модель вернула недостаточно вариантов, включается fallback‑догенерация программно.

### 4.3 Подсистема FAQ‑ассистента

Состоит из трёх частей:

1. импорт данных FAQ (`/support/faq/import`, `/support/dialogs/import`);
2. подбор релевантного контекста по токенному пересечению вопроса и FAQ‑пар;
3. генерация ответа в стиле техподдержки (`/support/faq/ask`).

Дополнительно сохраняются метрики качества: количество совпадений, средний/максимальный score релевантности, доля zero‑match.

### 4.4 Подсистема шаблонной генерации PHP‑страниц

`/page-template/generate-file` берёт шаблон из `templates/pages`, просит модель вернуть JSON по placeholder‑ключам и собирает готовый PHP‑файл на скачивание. Реализация защищает от path traversal через нормализацию имени шаблона.

### 4.5 Gateway‑подсистема

В gateway реализованы:

- регистрация/вход пользователя;
- выдача и использование `X-Gateway-Key`;
- каталог моделей (локальные + внешние);
- генерация через `/gateway/generate`;
- журнал потребления токенов;
- админ‑операции над пользователями и моделями;
- OpenAI‑совместимые эндпоинты `/v1/models` и `/v1/chat/completions`.

Это делает систему интегрируемой со сторонними клиентами, ожидающими OpenAI API‑формат.

## 5. Схема данных

Ключевые сущности:

- `RequestLog` — первичный журнал всех генераций;
- `SupportFaqEntry` — база FAQ;
- `SupportFaqQueryMetric` — аналитика качества FAQ;
- `GatewayUser` — пользователи кабинета;
- `GatewayModel` — внутренний каталог моделей и цен;
- `GatewayUsageLog` — фактическое потребление токенов по пользователям/моделям.

Такая схема позволяет в дипломе показать не только функциональность, но и управленческую аналитику (стоимость, активность, стабильность).

## 6. Верификация и экспериментальная часть

Проект уже содержит:

- автотесты `tests/test_app.py`;
- скрипт общего бенчмарка `scripts/benchmark_api.py`;
- скрипт сравнительного бенчмарка моделей `scripts/benchmark_gateway_models.py`;
- методологию эксперимента и чек‑листы в `docs/`.

Для диплома рекомендуется формализовать:

1. средние задержки и p95/p99;
2. распределение HTTP‑кодов;
3. среднее токенопотребление по моделям;
4. корреляцию нагрузки с `/stats`.

## 7. Что обязательно доработать перед защитой (приоритет P0)

1. **Добавить ограничение частоты запросов (rate limiting)** для публичных и gateway‑эндпоинтов.
2. **Усилить безопасность аутентификации**: добавить JWT‑сессию поверх API‑ключа для UI, ротацию ключей и политику истечения.
3. **Сделать централизованный обработчик ошибок** (единый JSON‑формат + коды ошибок + trace id).
4. **Разнести синхронные блокирующие вызовы** внешних API в фон/асинхронный слой, чтобы избежать деградации под нагрузкой.
5. **Добавить индексы и архивирование логов**, чтобы таблицы `request_logs` и `gateway_usage_logs` не деградировали при росте данных.
6. **Добавить role‑based policy в явном виде** (матрица прав, проверка на каждый admin‑endpoint).

## 8. Что можно улучшить (приоритет P1)

1. **Качество FAQ‑поиска**: перейти с token overlap на гибрид BM25 + векторный поиск (pgvector).
2. **Наблюдаемость**: Prometheus‑метрики, latency histogram, error budget, Grafana‑дашборды.
3. **Кеширование** повторяющихся запросов (Redis) для снижения стоимости и latency.
4. **Ограничение стоимости генерации**: квоты на пользователя/роль и hard limits по токенам.
5. **Расширение тестов**:
   - негативные сценарии безопасности;
   - тесты конкурентной нагрузки;
   - snapshot‑тесты контрактов OpenAI‑совместимого API.
6. **CI/CD**: автоматический запуск `pytest`, линтеров и миграций в pipeline.

## 9. Что можно вынести как научную новизну / практическую ценность

В тексте диплома можно подчеркнуть:

- единый gateway‑слой для локальных и внешних LLM с унифицированным API;
- практическая модель оценки стоимости токенов и журналирования usage;
- прикладная адаптация ИИ под узкий домен (хостинг/домены + FAQ + шаблонная генерация);
- воспроизводимая методика бенчмаркинга и проверяемые метрики качества эксплуатации.

## 12. Важные замечания для согласованности текста диплома

1. В проекте режим `php_page` внутри `/mode/run` **отключен**, и это нужно писать явно: генерация PHP выполняется отдельным эндпоинтом `POST /page-template/generate-file`.
2. В тексте отчёта важно разделять:
   - «базовый API» (`/generate`, `/generate/domains`, `/support/...`),
   - «gateway API» (`/gateway/...`),
   - «OpenAI‑совместимый слой» (`/v1/...`).
3. Формулировки «4 режима» из старых документов нужно аккуратно синхронизировать с текущей реализацией.
