> Раздел написан под фактическую реализацию проекта и ваши реальные результаты замеров от **19 апреля 2026 года**.

---

## 1. Цель тестирования

Цель тестирования — подтвердить, что система `ai-servise`:

1. корректно реализует функциональные требования API;
2. устойчиво работает в пользовательских и административных сценариях;
3. сохраняет данные в БД и отдает ожидаемую статистику;
4. обеспечивает приемлемые показатели производительности и стабильности по разным моделям.

Тестирование выполнено в двух контурах:

- **функциональная верификация** автотестами `pytest`;
- **нагрузочно-производительные замеры** скриптами `benchmark_api.py` и `benchmark_gateway_models.py`.

---

## 2. Методика функционального тестирования (pytest)

### 2.1 Инструменты и подход

- Фреймворк: `pytest`.
- HTTP-проверки: `fastapi.testclient.TestClient`.
- Модель в тестах: подменяется `FakeClient`, чтобы тесты были детерминированными и не зависели от реальных сетевых вызовов к LLM.
- БД для тестов: SQLite (`sqlite:////tmp/ai_servise_test.db`).

Таким образом, автотесты проверяют в первую очередь **контракты API, валидацию, маршрутизацию логики и запись данных**, а не качество генерации реальной LLM.

### 2.2 Какие подсистемы покрыты тестами

Автотесты покрывают следующие блоки:

1. **Доступность сервиса**
   - `test_health` — проверка `/health`, статуса базы и заголовка `X-Request-ID`.

2. **Базовая генерация и история**
   - `test_generate_and_history` — корректность `/generate` и последующей выборки `/history`.

3. **Потоковая генерация**
   - `test_generate_stream` — формат `application/x-ndjson`, наличие chunk-данных.

4. **Доменная генерация**
   - `test_generate_domains` — структура ответа, количество и формат доменов.

5. **Агрегированная статистика**
   - `test_stats` — ключевые поля `/stats` и их базовая корректность.

6. **Унифицированный режим `/mode/run`**
   - `test_mode_chat`, `test_mode_domains_list`, `test_mode_php_page_removed_from_mode_runner`.

7. **Шаблонная генерация файла**
   - `test_page_template_generate_file` — тип контента, attachment-выдача, имя файла.

8. **FAQ-ветка**
   - `test_faq_import_and_support_mode`, `test_support_dialog_import`, `test_support_faq_relevance_limit`.

9. **Безопасность импорт-эндпоинтов FAQ**
   - `test_admin_api_key_protects_import_endpoints` — проверка `401` без ключа и `200` при корректном `X-API-Key`.

Итог: тесты проверяют не только happy-path, но и критичные негативные сценарии доступа.

### 2.3 Команда запуска

```bash
python3 -m pytest -q
```

### 2.4 Что фиксировать в отчете

- общее число тестов;
- статус выполнения (passed/failed);
- дата и окружение запуска;
- версия Python (при наличии в протоколе запуска).

## 3. Метрики производительности: что именно измеряется

В benchmark-скриптах используются следующие метрики:

1. **`status_codes`**
   - распределение HTTP-кодов ответов.
   - ключевой индикатор стабильности API (доля `200`, наличие `4xx/5xx`).

2. **`success_rate_percent`** (в gateway benchmark)
   - доля успешных ответов среди всех замеренных запросов.
   - вычисляется как `(успешные / общее число) * 100%`.

3. **`latency_ms`**
   - `min` — минимальная задержка;
   - `avg` — средняя задержка;
   - `median` — медиана;
   - `p95` — 95-й процентиль (ориентир пользовательской стабильности);
   - `p99` — 99-й процентиль (хвостовая деградация);
   - `max` — максимальная задержка.

4. **`response_size_bytes.avg`**
   - средний размер ответа в байтах;
   - косвенно отражает «тяжесть» ответов и overhead передачи.

5. **`tokens`** (в gateway benchmark)
   - `avg_prompt_tokens`;
   - `avg_completion_tokens`;
   - `avg_total_tokens`.
   Эти метрики критичны для оценки себестоимости и тарификации gateway.

6. **`errors`**
   - список сообщений об ошибках при неуспешных запросах;
   - нужен для анализа отказов в конкретных моделях.

## 5. Сравнительный анализ результатов

### 5.1 По скорости

1. Самая быстрая по `avg latency`: **local/qwen2.5-3b** (~5.22 с).
2. Далее: **local/llama3.2-3b** (~7.84 с).
3. Самая медленная в вашем замере: **proxy/openrouter-deepseek-chat** (~8.72 с).

По хвостам (`p95/p99`) картина аналогичная: наибольшая вариативность задержки у внешней proxy-модели.

### 5.2 По объему ответа

- Наиболее «тяжелый» ответ у `local/llama3.2-3b` (в среднем 2630.6 байт), что коррелирует с более высоким completion tokens.
- Самый компактный ответ — у `proxy/openrouter-deepseek-chat` (1115.05 байт).

### 5.3 По токенопотреблению

- Максимальный `avg_total_tokens`: `local/llama3.2-3b` (371.75).
- Минимальный `avg_total_tokens`: `proxy/openrouter-deepseek-chat` (205.0).

Для раздела экономики/себестоимости это значит:

- `llama3.2-3b` потенциально дороже в токенах;
- `openrouter-deepseek-chat` при меньшем токенопотреблении имеет больший latency в вашем окружении.

### 5.4 По надежности

- Все три модели показали 100% успешность и отсутствие ошибок (`errors: []`).
- Это сильный аргумент для защиты: система стабильна в многомодельном режиме.

## 8. Шаблон текста вывода для диплома

По итогам функционального тестирования и нагрузочных экспериментов подтверждена корректность реализации API-сервиса `ai-servise` и его устойчивость в многомодельном режиме. Набор автотестов покрывает ключевые пользовательские и административные сценарии, включая проверку авторизации, потоковой генерации, FAQ-подсистемы и шаблонной генерации файла. Экспериментальные измерения показали 100% успешность ответов во всех исследуемых конфигурациях моделей. При этом минимальную среднюю задержку показала локальная модель `local/qwen2.5-3b`, максимальное токенопотребление — `local/llama3.2-3b`, а внешняя proxy-модель продемонстрировала наибольшие хвостовые задержки при стабильной успешности. Полученные результаты позволяют считать систему готовой к демонстрации на защите и дальнейшему развитию в части оптимизации latency и себестоимости генерации.

## 9. Что можно доусилить в следующей итерации тестирования

1. Добавить отдельные сценарии soak-теста (длительный прогон 1–2 часа).
2. Добавить конкурентную нагрузку (N параллельных клиентов).
3. Добавить негативные тесты gateway-admin API (гранулярные 401/403 кейсы).
4. Автоматизировать построение графиков из benchmark JSON (например, через pandas/matplotlib).
