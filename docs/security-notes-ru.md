# Примечания по безопасности и изоляции данных

Этот документ объясняет, как в проекте `ai-servise` устроена изоляция данных компании при использовании внешней модели (OpenRouter).

## 1) Какие режимы потенциально чувствительные

К чувствительным режимам относятся:

- `support_faq` (может содержать клиентские кейсы и внутренние регламенты),
- `php`-генерация по шаблонам (`/page-template/generate-file`),
- корпоративные сценарии через `/mode/run`.

Именно для них важно контролировать, в какую модель уходит запрос.

## 2) Текущая политика по умолчанию

По умолчанию внутренние режимы работают **только с локальными моделями Ollama**.

Добавлено ограничение:

- если для `chat/domains/support_faq/php_template` передать внешнюю модель (`provider=openai`), API вернет `HTTP 400`.

Это исключает случайную отправку внутреннего контента во внешний провайдер.

## 3) Когда внешние модели можно включить

Есть флаг окружения:

```env
ALLOW_EXTERNAL_IN_INTERNAL_MODES=false
```

- `false` (рекомендуется для закрытого контура) — внешний провайдер запрещен для внутренних режимов;
- `true` — разрешить внешние модели в этих режимах осознанно.

## 4) Что реально уходит в OpenRouter

Во внешний провайдер отправляется только то, что уходит через модели с `provider=openai`
(например, `proxy/openrouter-deepseek-chat` в gateway).

Передача выполняется через `call_openai_proxy()`:

- `Authorization: Bearer <OPENAI_API_KEY>`
- `Content-Type: application/json`
- payload с `model`, `messages`, `temperature`, `max_tokens`.

Отдельные заголовки сайта (`HTTP-Referer`, `X-Title`) не используются.

## 5) Почему «сайт» не мешает изоляции

UI-страницы gateway — это только клиент к вашему API.
Данные не отправляются «на сайт OpenRouter» напрямую из браузера.

Маршрут передачи данных такой:

1. Браузер -> ваш FastAPI;
2. FastAPI решает, какая модель вызовется;
3. Только если выбрана внешняя модель и это разрешено политикой, FastAPI делает серверный запрос в OpenRouter.

Следовательно, изоляция контролируется на вашем backend, а не на уровне фронтенда.

## 6) Что показать преподавателю как доказательство

1. Запрос с внешней моделью в `mode/run`:

```bash
curl -X POST "http://127.0.0.1:8080/mode/run" \
  -H "Content-Type: application/json" \
  -d '{"mode":"chat","model_id":"proxy/openrouter-deepseek-chat","payload":{"prompt":"test"}}'
```

Ожидаемо: `HTTP 400` с сообщением про ограничение на локальные модели.

2. Запрос с локальной моделью:

```bash
curl -X POST "http://127.0.0.1:8080/mode/run" \
  -H "Content-Type: application/json" \
  -d '{"mode":"chat","model_id":"local/qwen2.5-3b","payload":{"prompt":"test"}}'
```

Ожидаемо: успешный ответ.

3. Показать `.env` с:

```env
ALLOW_EXTERNAL_IN_INTERNAL_MODES=false
```

Это фиксирует политику «закрытый сервис».

## 7) Дополнительные меры для полного закрытого контура

Если у компании строгий запрет на внешние LLM:

1. Оставьте в каталоге только локальные модели (`provider=ollama`).
2. Деактивируйте/удалите `proxy/openrouter-deepseek-chat` в `/gateway/admin`.
3. Не задавайте `OPENAI_API_KEY` в production `.env`.
4. Оставьте:

```env
ALLOW_EXTERNAL_IN_INTERNAL_MODES=false
```

Тогда даже при ошибке пользователя внутренние режимы не отправят данные наружу, а gateway-вызовы внешней модели будут недоступны.
